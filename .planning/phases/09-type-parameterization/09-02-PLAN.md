**Abstract config hierarchy:**

Change the abstract types to carry T:
```julia
abstract type AbstractConfig{D<:AbstractDomain, T<:AbstractFloat} end
abstract type AbstractLiouvConfig{D<:AbstractDomain, T<:AbstractFloat} <: AbstractConfig{D,T} end
abstract type AbstractThermalizeConfig{D<:AbstractDomain, T<:AbstractFloat} <: AbstractConfig{D,T} end
```

IMPORTANT: This changes the type parameter count from 1 to 2. All existing dispatch signatures like `config::AbstractConfig{D}` must still work. In Julia, `AbstractConfig{D}` is shorthand for `AbstractConfig{D,T} where T`, so adding a second parameter is backward compatible for dispatch as long as existing code uses `AbstractConfig{D}` without specifying T. Verify this.

**LiouvConfig{D,T}:**
```julia
@kwdef struct LiouvConfig{D<:AbstractDomain, T<:AbstractFloat} <: AbstractLiouvConfig{D,T}
    num_qubits::Int64
    with_coherent::Bool
    with_linear_combination::Bool
    domain::D
    beta::T
    sigma::T
    gaussian_parameters::Union{Tuple{T, T}, Tuple{Nothing, Nothing}} = (nothing, nothing)
    a::Union{T, Nothing} = nothing
    b::Union{T, Nothing} = nothing
    num_energy_bits::Union{Int, Nothing} = nothing
    t0::Union{T, Nothing} = nothing
    w0::Union{T, Nothing} = nothing
    eta::Union{T, Nothing} = nothing
    num_trotter_steps_per_t0::Union{Int, Nothing} = nothing
end
```

Add a convenience constructor that accepts `precision` kwarg:
```julia
function LiouvConfig(; precision::Type{T}=Float64, kwargs...) where {T<:AbstractFloat}
    # Convert float kwargs to T
    LiouvConfig{...}(; converted_kwargs...)
end
```

The exact implementation: Julia's @kwdef generates a keyword constructor. For the `precision` kwarg pattern, the cleanest approach is:
- Keep `@kwdef` for the struct with `T` in float fields
- Julia infers `T` from the `beta::T` field value. So `LiouvConfig(beta=1.0, ...)` gives `T=Float64`, and `LiouvConfig(beta=Float32(1.0), ...)` gives `T=Float32`.
- Add a thin wrapper that accepts `precision=Float32` and converts: it takes all float kwargs, converts them to T, and forwards.
- For backward compat: `LiouvConfig(beta=1.0, ...)` just works (T=Float64 inferred).

Repeat this pattern for `LiouvConfigGNS{D,T}`, `ThermalizeConfig{D,T}`, `ThermalizeConfigGNS{D,T}`.

For GNS structs (manual constructors, not @kwdef): Update the inner constructor and keyword constructor to carry T. The `with_coherent` enforcement stays.

**LindbladianWorkspace{T}:**
```julia
struct LindbladianWorkspace{T<:AbstractFloat}
    Id::Matrix{Complex{T}}
    jump_tmp::Matrix{Complex{T}}
    jump_conj::Matrix{Complex{T}}
    jump_dag_jump::Matrix{Complex{T}}
    jump2_jump1::Matrix{Complex{T}}

    function LindbladianWorkspace{T}(dim::Int) where {T<:AbstractFloat}
        CT = Complex{T}
        Id = Matrix{CT}(I, dim, dim)
        jump_tmp = zeros(CT, dim, dim)
        jump_conj = zeros(CT, dim, dim)
        jump_dag_jump = zeros(CT, dim, dim)
        jump2_jump1 = zeros(CT, dim, dim)
        new{T}(Id, jump_tmp, jump_conj, jump_dag_jump, jump2_jump1)
    end
end
LindbladianWorkspace(dim::Int) = LindbladianWorkspace{Float64}(dim)
```

**JumpOp:**

Currently `JumpOp{T <: AbstractMatrix{ComplexF64}}`. Change to:
```julia
struct JumpOp{M <: AbstractMatrix{<:Complex}}
    data::M
    in_eigenbasis::Matrix{<:Complex}
    orthogonal::Bool
    hermitian::Bool
end
```
Or more precisely, parameterize on the float type:
```julia
struct JumpOp{T<:AbstractFloat, M<:AbstractMatrix{Complex{T}}}
    data::M
    in_eigenbasis::Matrix{Complex{T}}
    orthogonal::Bool
    hermitian::Bool
end
```
This is more type-stable. But it changes the type parameter meaning. Simpler approach: just widen the existing parameter to accept any complex type:
```julia
struct JumpOp{T <: AbstractMatrix{<:Complex}}
    data::T
    in_eigenbasis::Matrix{<:Complex}
    orthogonal::Bool
    hermitian::Bool
end
```
Choose whichever is cleanest. The key constraint: existing code constructs `JumpOp(data, eigvecs' * data * eigvecs, orth, herm)` where data is `Matrix{ComplexF64}`, so this must still work.
