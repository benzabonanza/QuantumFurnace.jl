Update `_gibbs_in_eigen` to be generic:
```julia
function _gibbs_in_eigen(eigvals::Vector{T}, beta::T) where {T<:AbstractFloat}
    dim = length(eigvals)
    CT = Complex{T}
    Z = sum(exp.(-beta .* eigvals))
    rho = zeros(CT, dim, dim)
    for i in 1:dim
        rho[i, i] = CT(exp(-beta * eigvals[i]) / Z)
    end
    return rho
end
```

Update the three HamHam constructors:

1. **HamHam(terms, coeffs, num_qubits, beta; periodic, hermitian_check, precision)**: The main constructor from terms/coefficients. Add `precision::Type{T}=Float64` kwarg. When `precision=Float32`:
   - Check input terms are `Matrix{ComplexF32}` or promotable (Float32 data to Float64 is OK; Float64 data to Float32 throws error)
   - Convert intermediate computations to use T/Complex{T}
   - Implementation: For backward compat, keep the existing Float64 path working unchanged. When precision differs from input, validate the mixed-precision policy.

2. **HamHam(terms, coeffs, disordering_terms, disordering_coeffs, num_qubits, beta; ...)**: Same pattern with `precision` kwarg.

3. **HamHam(raw::NamedTuple, beta)**: Infers T from `eltype(raw.eigvals)`. No precision kwarg needed -- this is an internal constructor used by BSON loader.

Mixed-precision policy (from CONTEXT.md locked decisions):
- If caller passes `precision=Float32` but input data is Float64: throw `ArgumentError("Expected Float32 data, got Float64. Reconstruct with Float32 inputs or use default Float64 precision.")`
- If caller passes default `precision=Float64` but input data is Float32: silently promote via `convert`.

Update `construct_base_ham`, `construct_disordering_terms`, `rescaling_and_shift_factors` to accept the generic types (these currently hardcode `ComplexF64` and `Float64`). Make them generic on T:
- `construct_base_ham` return type: `Hermitian{Complex{T}, Matrix{Complex{T}}}`
- The internal sparse computation can stay as `ComplexF64` intermediates for Float64 (the critical path) -- for Float32, convert at the end.
- `rescaling_and_shift_factors(hamiltonian::Hermitian)` -- already mostly generic, just remove explicit Float64 type annotations.

Also update `create_bohr_dict` if it exists in this file or in the file where it's defined (check -- it may be in energy_domain.jl or bohr_domain.jl). It needs to accept `Matrix{T}` instead of `Matrix{Float64}`.

Also update `find_ideal_heisenberg` to accept a `precision` kwarg and return a NamedTuple with fields of the appropriate type. Default remains Float64.
