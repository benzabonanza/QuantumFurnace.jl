<task type="auto">
  <name>Task 1: Simplify TrajectoryFramework type parameters</name>
  <files>src/trajectories.jl</files>
  <action>
**Investigation step**: Analyze which of the 5 type parameters `{T,C,H,PD,D}` are used for dispatch or performance:

- `T` (element type, currently ComplexF64): Used in `step_along_trajectory!` dispatch signatures (`TrajectoryFramework{ComplexF64,...}`) and in `PerOperatorKraus{T}`, `TrajectoryWorkspace{T}`. This is essential for type stability and will become more important in Phase 9 (type parameterization). **KEEP**.
- `C` (config type): NOT used for dispatch anywhere. The config is accessed at runtime via `fw.config` but never dispatched on through the TrajectoryFramework type parameter. **REMOVE**.
- `H` (ham_or_trott type): NOT used for dispatch anywhere. It's accessed as `fw.ham_or_trott` but only in the `EnergyDomain` variant of `step_along_trajectory!` (with an `@assert ham isa HamHam`). **REMOVE**.
- `PD` (precomputed_data type): NOT used for dispatch anywhere. Accessed as `fw.precomputed_data` at runtime. **REMOVE**.
- `D` (domain type): **Essential for dispatch** -- `step_along_trajectory!` dispatches on `D` to select EnergyDomain vs TimeDomain/TrotterDomain variants. **KEEP**.

**Result**: Reduce from `{T,C,H,PD,D}` to `{T,D}`.

**Step 1**: Update TrajectoryFramework struct definition:
```julia
struct TrajectoryFramework{T, D<:AbstractDomain}
    domain::D
    jumps::Vector{JumpOp}
    ham_or_trott::Union{HamHam, TrottTrott}
    config::AbstractThermalizeConfig{D}
    precomputed_data::Any  # NamedTuple from precompute_data, varies by domain

    # Per-operator Kraus data (Lie-Trotter splitting)
    per_operator::Vector{PerOperatorKraus{T}}
    n_jumps::Int

    # Step parameters
    delta::Float64
    delta_eff::Float64
    alpha::Float64

    # Runtime workspace (mutated during stepping)
    ws::TrajectoryWorkspace{T}
end
```

Note: `precomputed_data` becomes `Any` (or a typed NamedTuple -- but NamedTuples have different structures per domain, so `Any` is simplest). The `config` field uses `AbstractThermalizeConfig{D}` to tie the domain to the framework. `ham_or_trott` uses a Union type.

**Step 2**: Update `build_trajectoryframework` return type -- it already constructs with positional args, just remove the extra type params. The constructor call `TrajectoryFramework(config.domain, ...)` will infer `{ComplexF64, typeof(config.domain)}`.

**Step 3**: Update dispatch signatures in `step_along_trajectory!`:

Current:
```julia
function step_along_trajectory!(
    psi::Vector{ComplexF64},
    fw::TrajectoryFramework{ComplexF64,<:Any,<:Any,<:Any,D},
) where {D<:Union{TimeDomain,TrotterDomain}}
```

New:
```julia
function step_along_trajectory!(
    psi::Vector{ComplexF64},
    fw::TrajectoryFramework{ComplexF64,D},
) where {D<:Union{TimeDomain,TrotterDomain}}
```

And for EnergyDomain:
```julia
function step_along_trajectory!(
    psi::Vector{ComplexF64},
    fw::TrajectoryFramework{ComplexF64,EnergyDomain},
)
```

**Step 4**: Update `_evolve_along_trajectory!` signature:
```julia
function _evolve_along_trajectory!(
    psi::Vector{ComplexF64},
    fw::TrajectoryFramework{ComplexF64},
    ...
```
This one doesn't dispatch on D so it stays with just `{ComplexF64}` (or remove the explicit type constraint).
